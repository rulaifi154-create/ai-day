## Cursor 项目规则（alwaysApply）

> 说明：本文件中的规则默认 **alwaysApply（始终生效）**，适用于本 AI-Day 项目的所有对话与自动生成操作。

### 1. 语言与沟通
- **回复语言（alwaysApply）**  
  - 助手始终使用 **简体中文** 回复。
- **Markdown 结构（alwaysApply）**  
  - 标题只使用 `##` / `###`，**不要用 `#`**。  
  - 关键内容使用 **加粗**。  
  - 列表统一使用 `- ` 开头；「字段: 描述」风格统一写成 `- **字段**: 描述`。  
  - URL 必须用反引号（\`url\`）或 Markdown 链接 `[说明](url)` 包裹，禁止裸露 URL。

### 2. 代码与文档引用
- **引用已有代码（alwaysApply）**  
  - 当引用项目中已存在的代码时，必须使用 **代码引用块** 格式：

  ```text
  ```startLine:endLine:filepath
  // 这里是已有代码片段
  ```
  ```

  - `startLine`: 起始行号  
  - `endLine`: 结束行号  
  - `filepath`: 文件相对路径  
  - 不能添加语言标记（如 `ts`、`js`、`python` 等）。

- **书写/提议新代码（alwaysApply）**  
  - 当展示的是项目中尚不存在的新代码时，使用正常 Markdown 代码块：

  ```ts
  const x = 1;
  ```

- **禁止混合格式（alwaysApply）**  
  - 不允许同时使用「行号 + 文件路径」和「语言标记」的混合格式，二者必须分开使用。

### 3. 编辑代码/文件的工作流
- **默认允许修改仓库文件（alwaysApply）**  
  - 只要用户没有明确说「不要改代码/文件」，助手可以在项目里创建或修改文件（例如新建 PRD、说明文档等），并尽量直接落地到实际文件中。

- **大改动后检查 Lint（alwaysApply）**  
  - 对源码文件（如 `.ts` / `.tsx` / `.js` / `.py` 等）进行较大修改后，应调用 Lint 工具检查是否引入新的错误，并在可行范围内修复。
  - 对纯文档类文件（如 `.md`）通常不强制进行 Lint 检查。

- **Jupyter Notebook 特殊处理（alwaysApply）**  
  - 若使用 `.ipynb` 文件，只能通过专用的 `edit_notebook` 工具编辑。  
  - 每次只改一个 cell，且要保证 `old_string` 精确匹配原内容，避免误改。

### 4. 任务与计划管理
- **复杂需求使用 Todo 拆分（alwaysApply）**  
  - 当用户需求包含多步骤或较复杂（例如「实现整个 AI-Day 网站前后端」）时，使用内置 `todo_write` 工具创建任务列表。  
  - 任务应拆分为清晰的步骤（如：需求整理 / 设计 / 开发 / 联调 / 文档等）。  
  - 每当完成某项任务，要及时标记为 `completed`，保证不会遗漏。

### 5. 其他技术约束
- **禁止生成超长哈希或二进制内容（alwaysApply）**  
  - 不要生成极长的随机字符串、大块 base64、或其他二进制/无意义内容，以避免浪费资源。

- **新建 Web App 的 UI 要求（alwaysApply）**  
  - 若在本项目中新建前端 Web 页面，应尽量做到：  
    - 界面现代、美观，交互清晰、易用；  
    - 在合适的场景提供基础说明文档（如 `README`）和依赖管理文件（如 `package.json`、`requirements.txt` 等）。
